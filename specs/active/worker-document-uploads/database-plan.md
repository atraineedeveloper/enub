# Database Plan - Worker Document Uploads

## Status

Draft - no migrations created yet.

## 1. Findings from codebase inspection

### 1.1 `workers` table

- Name: `public.workers`
- Primary key: `id`, type `bigint`, backed by an identity sequence (`GENERATED BY DEFAULT AS IDENTITY`), not UUID.
- Defined in [supabase/migrations/20260702000000_remote_schema.sql:335](supabase/migrations/20260702000000_remote_schema.sql#L335).
- Child tables already reference `workers.id` as `bigint` (e.g. `date_of_admissions`, `sustenance_plazas`).

### 1.2 `semesters` table

- Name: `public.semesters`
- Primary key: `id`, type `bigint`, identity sequence.
- Columns: `semester`, `school_year`.
- Defined in [supabase/migrations/20260702000000_remote_schema.sql:184](supabase/migrations/20260702000000_remote_schema.sql#L184).

### 1.3 Supabase client setup

- Single client in [src/services/supabase.js](src/services/supabase.js), created from `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`.
- No service-role usage in the frontend. All application traffic runs as `anon` (pre-login) or `authenticated` (post-login) roles under RLS.
- Local config: `supabase/config.toml`. Only one migration file exists today, `20260702000000_remote_schema.sql`, which is a schema dump — there is no prior migration history to chain against.

### 1.4 Existing RLS pattern

Every existing table uses **flat, non-owner-scoped** policies:

- `SELECT` is `USING (true)` for **all** roles (including `anon`) on almost every table, including `workers` itself (name, RFC, address, phone, email are currently world-readable to anyone with the anon key).
- `INSERT` / `UPDATE` / `DELETE` are mostly `WITH CHECK (true)` / `USING (true)`, sometimes restricted `TO "authenticated"` (e.g. `date_of_admissions`, `sustenance_plazas`), sometimes open to all (`schedule_assignments`, `groups`).
- There is **no row-ownership model anywhere** — no `user_id` column, no `auth.uid()` check, no per-row scoping.

### 1.5 Auth model

- Auth is a single flat login: [src/features/authentication/useUser.js](src/features/authentication/useUser.js) only checks `user?.role === "authenticated"`. There is no per-user role/permission distinction (no "trabajador" vs "dirección" claim anywhere).
- [src/ui/ProtectedRoute.jsx](src/ui/ProtectedRoute.jsx) gates the entire app behind one boolean: authenticated or not.
- There is **no mapping between `auth.users` and `workers`**. Nothing in the schema or app associates a logged-in account with a specific worker row.
- Practically, this app today is an internal admin/staff panel — "workers" are managed records, not accounts that log in themselves.

This directly matters for the spec's stated decisions ("Los trabajadores subirán sus propios documentos", "Dirección podrá revisar los documentos") — see [Unresolved questions](#6-unresolved-questions).

### 1.6 Storage

- One bucket is used today, `profile_pictures`, referenced in [src/services/apiWorkers.js](src/services/apiWorkers.js) via `supabase.storage.from("profile_pictures")` and `getPublicUrl`.
- **The `profile_pictures` bucket is not defined anywhere in the migrations.** There is no `insert into storage.buckets` and no `storage.objects` policy in `20260702000000_remote_schema.sql`. It must have been created manually (Studio/dashboard) against the remote project. This means `bunx supabase db reset` locally will **not** recreate it — a real environment gap, not something to silently replicate.
- The existing bucket is treated as public (`getPublicUrl`, no signed URLs), appropriate for profile photos but not for identity documents.

### 1.7 Workers feature structure

- Data access: `src/services/apiWorkers.js` (plain functions, no repository class), `src/services/apiSemesters.js`.
- UI: `src/features/workers/*` (`WorkerTable.jsx`, `WorkerRow.jsx`, `CreateEditWorkerForm.jsx`, `useCreateWorker.js`, `useEditWorker.js`, `useWorkers.js`), page at `src/pages/Records/Workers.jsx`.
- Row actions use a shared `Menus` component (`src/ui/Menus.jsx`) — `WorkerRow.jsx` currently has one action, "Editar" ([src/features/workers/WorkerRow.jsx:56-62](src/features/workers/WorkerRow.jsx#L56-L62)). A "Documentos" action would be added here later, in Phase 5 (UI), not now.
- Child-record replace pattern already established (`apiWorkers.js`): delete-all-then-insert for `date_of_admissions` and `sustenance_plazas`, not upsert-by-key. Useful precedent for how "replace" is conventionally implemented in this codebase.

## 2. Proposed database schema

Three new tables, all using `bigint identity` primary keys to match `workers`/`semesters` convention.

### 2.1 `worker_document_categories`

Reference/lookup table, seeded once via migration.

| Column       | Type                                 | Notes                                              |
| ------------ | ------------------------------------ | -------------------------------------------------- |
| `id`         | `bigint identity`                    | PK                                                 |
| `name`       | `text not null`                      | e.g. "Datos personales", "Docencia"                |
| `scope`      | `text not null`                      | `'permanent'` or `'semester'`, enforced by `CHECK` |
| `sort_order` | `smallint not null default 0`        | display ordering                                   |
| `created_at` | `timestamptz not null default now()` |                                                    |

`UNIQUE (name)`.

### 2.2 `worker_document_types`

Reference/lookup table, seeded once via migration.

| Column            | Type                                                        | Notes                                       |
| ----------------- | ----------------------------------------------------------- | ------------------------------------------- |
| `id`              | `bigint identity`                                           | PK                                          |
| `category_id`     | `bigint not null references worker_document_categories(id)` |                                             |
| `name`            | `text not null`                                             | e.g. "CURP", "Evidencias"                   |
| `allows_multiple` | `boolean not null default false`                            | `true` only for the Evidencias-type entries |
| `sort_order`      | `smallint not null default 0`                               |                                             |
| `created_at`      | `timestamptz not null default now()`                        |                                             |

`UNIQUE (category_id, name)`. Index on `category_id`.

### 2.3 `worker_documents`

Uploaded file records — the only table written to at runtime.

| Column             | Type                                                       | Notes                                                                         |
| ------------------ | ---------------------------------------------------------- | ----------------------------------------------------------------------------- |
| `id`               | `bigint identity`                                          | PK                                                                            |
| `worker_id`        | `bigint not null references workers(id) on delete cascade` |                                                                               |
| `document_type_id` | `bigint not null references worker_document_types(id)`     |                                                                               |
| `semester_id`      | `bigint references semesters(id)`                          | `NULL` for permanent-scope categories, required for semester-scope categories |
| `file_name`        | `text not null`                                            | original file name                                                            |
| `storage_path`     | `text not null`                                            | path inside the storage bucket, globally unique                               |
| `mime_type`        | `text not null`                                            |                                                                               |
| `file_size`        | `bigint not null`                                          | bytes, enforced with `CHECK (file_size > 0 AND file_size <= 10485760)`        |
| `uploaded_by`      | `uuid references auth.users(id)`                           | nullable — populated from `auth.uid()` at insert time when available          |
| `created_at`       | `timestamptz not null default now()`                       | upload date                                                                   |

Constraints:

- `UNIQUE (storage_path)` prevents two metadata rows from pointing to the same storage object.
- `CHECK (file_size > 0 AND file_size <= 10485760)` enforces a positive file size and the 10 MB MVP limit.

Indexes: `(worker_id)`, `(worker_id, semester_id)`, `(document_type_id)`.

### 2.4 Data-integrity triggers (confirmed)

Two invariants can't be expressed as plain `CHECK`/`UNIQUE` constraints because they depend on a joined table. Both are implemented as database triggers, per [decisions.md #12](specs/active/worker-document-uploads/decisions.md):

1. **Scope consistency** — a `BEFORE INSERT OR UPDATE` trigger on `worker_documents` that looks up the document type's category `scope` and rejects the row if `scope = 'permanent'` and `semester_id IS NOT NULL`, or `scope = 'semester'` and `semester_id IS NULL`.
2. **Single active file for non-Evidencias types** — a `BEFORE INSERT` trigger that, when the target `document_type.allows_multiple = false`, rejects the insert if a row already exists for the same `(worker_id, document_type_id, semester_id)`.

Both triggers protect the invariant regardless of which client writes the row (app, Studio, future scripts, concurrent requests), not just the current UI.

**Replacement stays app-layer.** For single-file document types, `apiWorkerDocuments.js` (Phase 3, not built yet) deletes the old storage object and the old row first, then inserts the replacement — matching the existing `replaceWorkerDateOfAdmissions` delete-then-insert pattern. The trigger's role is to reject accidental duplicate rows (e.g. a race between two concurrent uploads), not to perform the replace itself.

### 2.5 No table for reports

The worker document status report (task list Phase 6) is a read-only aggregation of `worker_documents` joined against `worker_document_categories`/`worker_document_types`/`workers`/`semesters` for a single worker. No dedicated report table is needed — it can be produced by a query in `apiWorkerDocuments.js` and rendered/exported client-side, consistent with how `WorkerSheetSemester.jsx` already generates a PDF client-side from queried data.

## 3. Storage bucket plan

- New bucket: **`worker_documents`**, distinct from `profile_pictures`.
- **Confirmed `public = false`** ([decisions.md #13](specs/active/worker-document-uploads/decisions.md)), deviating intentionally from `profile_pictures`, which is public. Documents include CURP, INE/credencial de elector, constancia fiscal — personal identity documents, not photos. View/download goes through `createSignedUrl`; `getPublicUrl` is not used for this bucket.
- `file_size_limit`: 10 MB, matching the spec's stated max.
- `allowed_mime_types`: restrict at the bucket level in addition to client-side validation, covering PDF, Word (`.doc`/`.docx`), Excel (`.xls`/`.xlsx`), and image types (`.jpg`/`.jpeg`/`.png`/`.webp`).
- Bucket creation **is** representable in SQL (`insert into storage.buckets (...)`) and should be added as a migration, unlike `profile_pictures` which currently has no migration record at all.
- Path convention: `{worker_id}/{document_type_id}/{semester_id | 'permanent'}/{uuid}-{original_filename}`, mirroring the `worker-${crypto.randomUUID()}.${extension}` naming already used for profile pictures, but scoped into folders so a worker's files aren't a flat namespace.

## 4. RLS policy plan

Confirmed staff-facing access model ([decisions.md #11](specs/active/worker-document-uploads/decisions.md)): any authenticated internal user may upload or view documents for any worker in this MVP; no per-worker or per-uploader row scoping. Proposed policies follow the existing project convention (permissive by role, not by row owner), but are **stricter than the existing `workers` table** because these rows reference sensitive personal documents:

| Table                        | SELECT                                                                                | INSERT                               | UPDATE                                                                                                                                  | DELETE                          |
| ---------------------------- | ------------------------------------------------------------------------------------- | ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| `worker_document_categories` | `USING (true)` — all roles (reference data, safe to expose)                           | none (seeded via migration only)     | none                                                                                                                                    | none                            |
| `worker_document_types`      | `USING (true)` — all roles                                                            | none (seeded via migration only)     | none                                                                                                                                    | none                            |
| `worker_documents`           | `TO authenticated USING (true)` — **not** open to `anon`, unlike most existing tables | `TO authenticated WITH CHECK (true)` | `TO authenticated USING (true) WITH CHECK (true)` (only needed if metadata edits are allowed; likely unused if replace = delete+insert) | `TO authenticated USING (true)` |

`storage.objects` policies for the `worker_documents` bucket: mirror the table policy — `SELECT`/`INSERT`/`UPDATE`/`DELETE` restricted `TO authenticated`, `bucket_id = 'worker_documents'`, no `anon` access. This differs from typical public-bucket policies and is necessary since the bucket is private.

This intentionally does **not** implement "workers only see/upload their own documents" or "only dirección can review" — that's deferred by design (Section 6).

`uploaded_by` is populated from `auth.uid()` at insert time and records the staff account that performed the upload, not the worker the document is about ([decisions.md #15](specs/active/worker-document-uploads/decisions.md)). `worker_id` is the sole identifier of whose document it is.

## 5. Seed data plan

Seed `worker_document_categories` and `worker_document_types` from [source.md](specs/active/worker-document-uploads/source.md):

- Datos personales (`scope = 'permanent'`): Acta de nacimiento, CURP, Curriculum Vitae actualizado, Credencial de elector, Constancia de situación fiscal (SAT), Nombramiento.
- Docencia (`scope = 'semester'`): Planeación semestral, Plan de trabajo semestral, Planeaciones semanales, Rúbricas, Listas de cotejo, Evidencias (`allows_multiple = true`), Listas de asistencia, Actas de evaluación, Concentrado de calificaciones finales.
- Tutoría (`scope = 'semester'`): Relación de estudiantes tutorados, Canalizaciones, Evidencias de actividades (`allows_multiple = true`), Listas de asistencia, Informes de tutoría.
- Asesoría (`scope = 'semester'`): Control de asesorías, Evidencias (`allows_multiple = true`), Documentos de titulación, Informes.
- Investigación (`scope = 'semester'`): Artículos publicados, Ponencias, Capítulos de libro, Informes técnicos, Productos académicos diversos (note: source.md has a typo, "diversoss" — should be corrected when seeding, not reproduced).

Delivered via a versioned migration (`insert into ... on conflict do nothing` or plain `insert` in a reset-safe migration), not `supabase/seed.sql`, so the reference data ships with the schema and survives `db push` the same way table structure does. `seed.sql` can stay reserved for local dev/demo data if any is added later.

## 6. Resolved decisions

All previously open questions were resolved by the stakeholder/developer — see `decisions.md` #11-#15 for full rationale:

1. **Identity mapping gap.** This MVP is staff-facing only. Any authenticated internal user may upload/view documents for any worker; no ownership-based restriction is implemented in this phase. ([decisions.md #11](specs/active/worker-document-uploads/decisions.md))
2. **Trigger-based invariants.** Confirmed — implement as DB triggers (Section 2.4). Replacement for single-file types stays app-layer (delete-then-insert). ([decisions.md #12](specs/active/worker-document-uploads/decisions.md))
3. **`worker_documents` bucket privacy.** Confirmed private (`public = false`) with `createSignedUrl` for view/download; `getPublicUrl` is not used. ([decisions.md #13](specs/active/worker-document-uploads/decisions.md))
4. **`profile_pictures` bucket migration gap.** Out of scope for this feature. `worker_documents` must be created via migration so it doesn't repeat the same gap; a separate follow-up spec should address `profile_pictures` later. ([decisions.md #14](specs/active/worker-document-uploads/decisions.md))
5. **`uploaded_by` semantics.** Records the staff/internal account that performed the upload, not the worker. Not made to equal the worker until self-service accounts exist. ([decisions.md #15](specs/active/worker-document-uploads/decisions.md))

## 7. Future follow-ups (separate spec, not part of this feature)

Deferred by the access-model decision (Section 6, item 1) — not to be implemented as part of worker document uploads:

- `auth.users` to `workers` mapping.
- Worker self-service portal (workers log in and manage only their own documents).
- Dirección/admin role tier, distinct from general staff.
- Ownership-based RLS policies once the above exist (replacing the current flat "any authenticated user" policies on `worker_documents` and its storage bucket).
- Making the `profile_pictures` bucket reproducible locally via migration (unrelated pre-existing gap, noted in Section 1.6).
