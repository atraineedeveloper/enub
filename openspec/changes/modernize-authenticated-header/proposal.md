## Why

The authenticated header (`src/ui/Header.tsx`) was designed when Enub only had administrator users: it shows a sidebar toggle, a centered "ENUB" wordmark, a dark-mode button, and an isolated logout icon with no label. Now teacher/worker accounts also authenticate through `WorkerAppLayout` and land on this same header, but it gives no identity or role context — no name, no avatar, no indication of which of the three account roles (`admin`, `staff`, `worker`) is active — and the bare logout icon is ambiguous and has no accessible name. As the user base diversifies, the header needs to communicate who is logged in and under what role, safely and deterministically, on both desktop and mobile.

## What Changes

- Replace the centered wordmark + bare icon-row header with a three-area layout: left (sidebar toggle when present, ENUB brand, optional subtitle), center/context (current-page label from a centralized, most-specific-match route resolver), right (theme toggle, account control). The left area no longer reserves an empty placeholder column when no sidebar toggle exists (fixes worker-layout alignment).
- Add `useCurrentIdentity()`, an explicit discriminated-state hook (`loading | ready | incomplete | denied | profile-error | worker-error`) composing `useProfile()` with a strictly-gated worker lookup, so the header renders a defined, minimal-safe UI for every state and never flashes a guessed name, role, or initials.
- Use exactly three visible role labels, sourced from `profiles.role`: `admin` → `Administrador`, `staff` → `Personal administrativo`, `worker` → `Docente`. `worker` remains the internal role identifier; `Docente` is UI-only terminology. `staff` and `admin` are never merged into one label.
- Gate the worker-record lookup strictly: it only runs when the profile query succeeded, `role === "worker"`, and `workerId` is a finite, positive integer. `admin`/`staff` sessions never trigger a worker fetch.
- A `worker`-role profile with a missing, zero, negative, non-integer, `NaN`, or infinite `workerId` resolves to `incomplete` — the minimal authenticated header only (brand, theme toggle, direct logout) — never a fallback identity, since the profile's worker claim has no valid linkage to resolve. This is distinct from a *valid* `workerId` whose lookup finds no row (still `ready`, using the safe email-local-part fallback identity) and from a *valid* `workerId` whose lookup fails at the transport/RLS/database level (`worker-error`, also minimal header, but distinguished from the missing-row case so a real backend failure is never silently presented as an ordinary missing-name fallback).
- Derive identity output directly from live query state (no component-local retention), and enforce an explicit current-user/profile-generation guard in the resolver: worker-query output is only treated as authoritative when it belongs to the exact authenticated user id, resolved profile, and `workerId` active in the current render — a stale snapshot from a previous account (even one that happens to share a numeric `workerId`) is rejected rather than rendered. Switching authenticated accounts immediately drops the previous user's name/role/avatar and enters a neutral state until the new profile resolves; the existing `["worker", workerId]` query key is unchanged.
- Add an `Avatar` component (extracted/generalized from the pattern already used in `WorkerRow.tsx`) with the fallback order: profile picture (from the existing trusted `getProfilePicturePublicUrl` helper only) → Unicode-safe initials → generic user icon. Initials are derived from the display name only after any email local-part extraction, so an email domain can never leak into initials.
- Replace the isolated logout icon with an accessible account **popover** (not full ARIA application-menu semantics): a real button trigger with `aria-expanded`/`aria-controls`, Tab/Shift+Tab navigation, and dismissal-specific focus behavior — Escape closes and returns focus to the trigger; outside click closes without forcibly moving focus; route navigation closes without returning focus to the old trigger; logout follows the existing flow with no separate focus-restoration attempt during its navigation — containing display name, role label, and a `Cerrar sesión` action that reuses the existing `useLogout()` flow. In every non-`ready` state the header instead shows a direct, always-available logout control (no popover, no identity trigger).
- Add a route-to-label resolver using `react-router-dom`'s `matchPath` against an ordered, most-specific-first pattern list (supports parameterized and nested routes), including the exact mapping `/semesters/:id` → "Horario del semestre" (distinct from `/semesters` → "Semestres"), never displaying a raw URL segment and falling back to `ENUB` for unmapped routes.
- Apply deterministic responsive collapsing using exactly two fixed viewport-width breakpoints (no container queries): the existing `900px` mobile breakpoint, plus one new, documented intermediate threshold at `1100px`. Full identity + context above `1100px`; subtitle hides between `901px`–`1100px` while route context becomes truncatable; at `≤900px` route context and visible name/role text hide as well, leaving only the menu toggle, compact brand, and the avatar/account trigger (or the direct logout control in minimal states). Interactive targets are at least 44×44 CSS px throughout.
- **BREAKING**: none — this is a visual/structural change to an existing component with no route, prop-contract, or Supabase schema changes to callers (`AppLayout` and `WorkerAppLayout` continue to render `<Header />`/`<Header onToggleSidebar={...} />`).

## Capabilities

### New Capabilities
- `authenticated-header`: The authenticated header's identity-state model, role-aware account popover, route-context label, avatar/initials fallback, worker-query gating, stale-account isolation, and responsive/accessibility behavior across the admin and worker layouts.

### Modified Capabilities
_None — no existing `openspec/specs/` capability currently covers header/navigation UI; this is a net-new capability, not a change to an existing spec's requirements._

## Impact

- **Code**: `src/ui/Header.tsx` (rewritten), new `src/ui/Avatar.tsx`, new `src/ui/AccountPopover.tsx`, new route-label resolver (e.g. `src/ui/routeContext.ts`), new `src/features/authentication/useCurrentIdentity.ts` (composes `useProfile()` + a strictly-gated worker lookup), `src/features/workers/useWorker.ts` (gating signature widened), a new worker-identity service function distinguishing "no row" from "query error" (see design.md), `src/ui/AppLayout.tsx` and `src/ui/WorkerAppLayout.tsx` (no structural change — same `Header` invocation), `src/features/authentication/Logout.tsx` usage moves into the direct logout control and the popover's logout action (same underlying `useLogout()` hook, no second implementation).
- **Data**: Read-only additional usage of the existing `workers` table (`name`, `profile_picture`) via a gated, dedicated worker-identity lookup, and `profiles` table (`role`, `worker_id`) via the existing `useProfile()` hook. No schema changes, no new Supabase tables.
- **Dependencies**: None added — reuses existing `styled-components`, `react-icons/hi2`, `react-router-dom`'s `matchPath`, `src/hooks/useOutsideClick.ts`, and the repository's existing `bun:test` setup (already used by `src/services/apiWorkerDocuments.test.ts` and `src/features/workers/documents/workerDocumentTypeVisibility.test.ts`) for pure-logic automated tests.
- **Routes**: No route changes. `AppLayout` and `WorkerAppLayout` keep their current route trees in `src/App.tsx`.
- **Migration**: None expected — this is a UI-only change to an existing shared component; both authenticated layouts already import `Header` from the same file, so no consumer needs to change how it's invoked beyond the existing `onToggleSidebar` prop.

## Final Decisions

These were previously open questions in `design.md`; they are now resolved and binding for this change:

- `staff` has its own visible label (`Personal administrativo`), distinct from `admin` (`Administrador`).
- Route context uses a centralized, most-specific-match resolver (`matchPath`-based, ordered patterns), not first-segment matching; `/semesters/:id` is exactly "Horario del semestre".
- The email local-part display name is an explicit, temporary fallback — not a long-term identity source; a future change may add a real display-name field. It applies to `admin`/`staff` sessions and to a `worker` session with a *valid* `workerId` whose row lookup finds nothing — never to a `worker` session with an invalid/missing `workerId`, which is `incomplete` instead.
- The `pending-access`/no-access state renders the minimal authenticated header (brand, theme toggle, direct logout), never a guessed identity.
- Account interaction uses an accessible popover (button semantics + `aria-expanded`/`aria-controls`), not full ARIA `role="menu"` application-menu semantics, with dismissal-specific focus rules (only Escape returns focus to the trigger).
- Responsive collapsing uses exactly two fixed viewport-width breakpoints (`900px` existing, `1100px` new) via ordinary media queries — not container queries and not a third breakpoint.
