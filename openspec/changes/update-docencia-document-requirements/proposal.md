## Why

The school director requested three catalog changes under the `Docencia` worker-document category: retire `Plan de trabajo semestral` and `Planeaciones semanales` from active use, and rename `Evidencias` to `Evidencias bimestrales`. `worker_document_types` currently has no lifecycle concept — every row is permanently visible in every upload interface, and the only way to "remove" a requirement today would be deleting or renaming its catalog row outright, which would either orphan or mislabel any `worker_documents` rows already uploaded against it. This change introduces a safe retirement model (an `is_active` flag) instead, so a retired requirement stops accepting new uploads without touching a single historical document or storage object.

## What Changes

- Add `worker_document_types.is_active boolean not null default true` via a new forward migration (the already-applied seed migration is never edited).
- A later migration (also new, not editing the seed) marks `Docencia / Plan de trabajo semestral` and `Docencia / Planeaciones semanales` inactive, and renames `Docencia / Evidencias` to `Evidencias bimestrales` in place (same row `id`, same `allows_multiple = true`, same `sort_order`) — scoped strictly to the `Docencia` category, never touching the identically-named `Asesoría / Evidencias` or `Tutoría / Evidencias de actividades` rows. The migration includes an explicit post-condition check that fails loudly rather than silently succeeding on an inconsistent catalog state (missing rows, an unexpected duplicate, or both the old and new `Evidencias` names present at once).
- **BREAKING (data-shape only, not a runtime break)**: any code or query that assumed every `worker_document_types` row is implicitly "active" must now read `is_active` explicitly. This change updates every in-repo consumer; no consumer is left silently relying on the old implicit assumption.
- Add database-level enforcement: a new, narrowly-scoped trigger (matching the existing `enforce_worker_document_scope`/`enforce_single_worker_document_file` pattern) rejecting any `INSERT` targeting an inactive `document_type_id`, and rejecting an `UPDATE` only when it *changes* `document_type_id` to an inactive one — an unrelated edit to an existing row that stays attached to the same (possibly inactive) type is never blocked. This is defense-in-depth against a genuine race, layered underneath a **service-layer preflight** (both `uploadWorkerDocument` and `replaceWorkerDocument` check `is_active` before any storage upload, failing fast with a controlled message for the ordinary stale-client case). Neither of these is a new authorization boundary — the existing ownership-aware RLS (staff/admin, or a worker scoped to their own `worker_id` via `current_worker_id()`) already governs who may act on a row at all; this change adds no new RLS policy and does not weaken the existing ones.
- Replace the current unsafe, non-transactional delete-before-insert replacement ordering with a new `SECURITY INVOKER` RPC (`replace_worker_document_metadata`) that validates input, verifies the type is active and single-file, deletes the superseded metadata row(s), and inserts the replacement row — all inside **one database transaction**. Because a PL/pgSQL function's statements share the transaction of the call that invoked it, any failure (an inactive-type rejection, the single-file integrity trigger, an RLS rejection, or any other database error) aborts the whole transaction and Postgres automatically rolls back everything the function did, including the delete — restoring the previous metadata with no custom compensating code. The existing `enforce_single_worker_document_file` trigger is never disabled or bypassed; deleting the old row before inserting the new one, in the same transaction, is what makes the insert compatible with that trigger in the first place.
- Update the shared document catalog consumption in `WorkerDocumentsView.tsx` (and the PDF report builder in `apiWorkerDocuments.ts`) so a document-type row is included for a given worker only when it is active **or** that same worker already has existing documents under it — never because any other worker has historical documents under it. Upload/replace controls are omitted for inactive types even when the historical row is shown; status is never "Pendiente" for an inactive type.
- Regenerate `src/types/supabase.ts` via the standard Supabase CLI type-generation command to reflect the new column.
- Add pgTAP coverage for the new column, the two retirements, the rename (including a capture-then-compare identity check that never hard-codes a row id), the narrowed trigger, the replacement RPC's transactional rollback behavior (inactive-type rejection, insert failure, RLS rejection all restore the previous metadata), the retirement-race schedule, single-file-trigger compatibility, and the untouched Tutoría/Asesoría rows; update the three existing pgTAP files that reference `Docencia / Evidencias` by exact name; add an improved read-only preflight query that always reports one row per expected type, existing or not, and doubles as the deployment-time identity capture-then-compare mechanism.

## Capabilities

### New Capabilities

- `worker-document-catalog-lifecycle`: the `is_active` lifecycle flag on `worker_document_types`, the retirement/rename data change under `Docencia`, server-side enforcement against uploads to inactive types, and the catalog-consumer behavior that keeps historical documents visible while hiding retired types from new uploads.

### Modified Capabilities

_None._ `openspec/specs/` has no existing capability covering the worker-document catalog; the only prior specs for this area live under `specs/active/worker-document-uploads/`, `specs/active/worker-self-service-documents/`, and `specs/active/worker-documents-ux-and-delete/` — all pre-existing, untouched, and out of scope for this change per project rules.

## Impact

- **Database**: `worker_document_types` (new column, new/updated rows), a new narrowly-scoped `BEFORE INSERT OR UPDATE` trigger and a new `replace_worker_document_metadata` RPC on/for `worker_documents`, no changes to `worker_documents`' own columns, existing rows, or its existing ownership-aware RLS policies (staff/admin full access; a worker scoped to `worker_id = current_worker_id()` — both additive-only, unchanged in shape).
- **Frontend**: `src/services/apiWorkerDocuments.ts` (`uploadWorkerDocument`/`replaceWorkerDocument` preflight + the new RPC-based replace flow, plus the report-status union rule), `src/features/workers/documents/WorkerDocumentsView.tsx` (row-inclusion union rule) — both the admin (`src/pages/Records/WorkerDocuments.tsx`) and worker self-service (`src/pages/MyDocuments.tsx`) routes render the same `WorkerDocumentsView`, so one component change covers both.
- **Documentation**: `docs/ai/api.md`, `docs/ai/architecture.md`, `docs/ai/testing.md` — each references `Evidencias` by name as a live example and needs updating to `Evidencias bimestrales`.
- **Generated types**: `src/types/supabase.ts` regenerated, not hand-edited.
- **Existing tests updated (not left to break)**: `worker_documents_seed.test.sql`, `worker_documents_triggers.test.sql`, `worker_documents_ownership_rls.test.sql` — all three reference `Docencia / Evidencias` by exact name today.
- **No changes** to `create-worker-account`, `resend-worker-access-link`, any Auth flow, storage bucket policies' shape, the existing ownership-aware RLS policies' shape, or any other document category (`Datos personales`, `Tutoría`, `Asesoría`, `Investigación`).
