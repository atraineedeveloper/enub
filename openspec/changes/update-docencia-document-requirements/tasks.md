## 5. Database-level safety: transactional replacement RPC

- [x] 5.1 Add a new migration creating `public.replace_worker_document_metadata(p_worker_id bigint, p_document_type_id bigint, p_semester_id bigint, p_file_name text, p_storage_path text, p_mime_type text, p_file_size bigint) RETURNS TABLE(new_id bigint, new_worker_id bigint, new_document_type_id bigint, new_semester_id bigint, new_file_name text, new_storage_path text, new_mime_type text, new_file_size bigint, new_uploaded_by uuid, new_created_at timestamptz, old_storage_paths text[])` exactly as specified in design.md Decision 5 (corrected, delete-then-insert): (1) validate all input is non-null; (2) **acquire the shared lifecycle advisory lock (Decision 14, finding #1) keyed by `p_document_type_id` before reading it**; (3) a plain `SELECT is_active, allows_multiple` on the target document type (no `FOR SHARE` — that table has only a SELECT-only RLS policy, and Postgres silently drops all rows from a locked SELECT when no policy applicable to the lock exists, discovered by exercising this RPC under a real simulated authenticated role in pgTAP), raising the default code if it does not exist; (4) raise `WDT01` if `is_active = false`; (6) raise if `allows_multiple = true` (replacement is single-file-only); (5)+(7) `FOR ... IN SELECT id, storage_path ... FOR UPDATE LOOP` over the existing worker/type/semester row(s) in `worker_documents` (which does have an applicable `FOR ALL` RLS policy, so this lock works correctly), capturing ids and storage paths into arrays (not `SELECT array_agg(...) ... FOR UPDATE`, which Postgres rejects); (8) `DELETE ... WHERE id = ANY(v_old_ids)`; (9) `INSERT ... RETURNING *` the replacement row **(finding #4: the full row, not just its id, so the client never needs a separate post-commit fetch)**; (10) `RETURN QUERY SELECT` every new-row column plus the captured old storage paths. Output columns are prefixed `new_`/`old_` specifically because `RETURNS TABLE` column names become implicitly-declared PL/pgSQL variables in the function body — an unprefixed `id` collided with the real `worker_documents.id` column and made every bare `WHERE id = ...` inside the function ambiguous (`42702`), caught by running the pgTAP suite after the first attempt. Mark the function `SECURITY INVOKER` explicitly (the default, stated for clarity).

FOR UPDATE` loop, the `DELETE`, and the `INSERT` exactly as it already governs direct client calls today.

- [x] 5.4 Confirm that a successful call returns the new row's complete data and an array of the superseded row(s)' storage paths, with the old row(s) actually removed from `worker_documents` and the new row present — both changes durable in the same committed transaction.
