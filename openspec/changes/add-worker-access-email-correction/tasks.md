## 3. Claim resolution RPC — trusted inputs, pre-claim validation, non-destructive reconciliation

- [ ] 3.1 Add migration creating `public.claim_worker_access_email_correction(worker_id bigint, requested_email text) RETURNS TABLE(operation_id bigint, outcome text, reason_code text)` — `SECURITY DEFINER`, `LANGUAGE plpgsql`, `SET search_path = ''`. No parameter for linked Auth user id, raw expected worker email, a pre-canonicalized email, or a profile id.
- [ ] 3.2 Canonicalize `requested_email` internally (`lower(trim(...))`); raise a stable-coded exception for an empty or malformed result (mapped by the Edge Function to `status: "invalid_email"`) — checked before any lock or row access.
- [ ] 3.3 Acquire the worker-identity lock using the corrected, single-string-argument `hashtextextended` form (design.md §7): `PERFORM pg_advisory_xact_lock(hashtextextended('worker_access_email:worker:' || worker_id::text, 0));`. Never a three-argument call anywhere in this migration or any other in this change.
- [ ] 3.4 Re-read the worker and its raw `email` after the lock; return `worker_not_found` if absent (no row created).
- [ ] 3.5 Resolve and validate the exact linked profile (`profiles WHERE worker_id = ... AND role = 'worker'`); return `worker_not_linked` if absent, `invalid_profile_role` if present with an unexpected role (no row created).
- [ ] 3.6 Derive the linked Auth user id from that profile; acquire the Auth-identity lock, same corrected form: `PERFORM pg_advisory_xact_lock(hashtextextended('worker_access_email:auth:' || linked_auth_user_id::text, 0));` — worker lock always first.
- [ ] 3.7 Re-read the profile's linked Auth id one more time under both locks; if it no longer matches what was just resolved, return `ambiguous_claim_state` with **no operation created** (a pre-claim race against an unrelated, non-lock-sharing RPC such as `link_worker_account`/`unlink_worker_account`).
- [ ] 3.8 **(Finding: verify the linked Auth user exists before claiming.)** Read `auth.users` directly for the linked Auth user id; if no such row exists, return `linked_auth_user_missing` — no operation created, no mutation.
- [ ] 3.9 **(Finding: reject a duplicate worker email before creating any claim.)** Check whether any *other* worker's `lower(trim(email))` already equals the canonical requested email; if so, return `duplicate_worker_email` — no operation created, no mutation. This is the identical check the sync RPC re-performs later inside its own transaction, as defense-in-depth against a race between this check and the eventual write.
- [ ] 3.10 Search for a blocking row (`state IN ('active', 'manual_attention_required')`) two ways, independently: by `worker_id` (`row_by_worker`) and by `linked_auth_user_id` (`row_by_auth`).
- [ ] 3.11 Reconcile — **never attempting an insert where a blocking row already exists for either identity:**
  - [ ] 3.11.1 Neither search finds anything → proceed to 3.12 (the only path that inserts).
  - [ ] 3.11.2 Both searches find the *same* row: that row is `manual_attention_required` → `outcome = 'manual_attention_blocking'`; `active` with a matching target → `outcome = 'resumed'`; `active` with a different target → `outcome = 'different_target_in_progress'`. No mutation, no second row, in any of these three cases.
  - [ ] 3.11.3 Both searches find *different* rows (a genuinely inconsistent, invariant-broken state) → `outcome = 'ambiguous_claim_state'`, `operation_id = NULL`, **do not touch either row**, do not attempt any insert, do not expose which row or constraint was involved.
  - [ ] 3.11.4 Exactly one search finds a row (the other finds nothing) — that row's own recorded identity disagrees with what was just freshly resolved: if that row is `active`, transition it **in place** (`UPDATE ... SET state = 'manual_attention_required', last_reason_code = 'ambiguous_claim_state'` — never an `INSERT`); if already `manual_attention_required`, leave it as-is; either way, `outcome = 'ambiguous_claim_state'`, `operation_id` = that row's id.
- [ ] 3.12 Only reached when 3.11.1 applies: check via direct SQL whether the freshly-read raw `workers.email` **and** a direct read of `auth.users.email` for the linked Auth user id both already canonically equal the requested email. If both: insert a new row **directly as `state = 'completed'`**, `outcome = 'already_completed'`. Otherwise: insert a new row as `state = 'active'`, `outcome = 'created'`.
- [ ] 3.13 Wrap the insert from 3.12 in an exception handler catching only `unique_violation`; use `GET STACKED DIAGNOSTICS ... constraint_name` to confirm it is one of this table's own two partial indexes before re-running 3.10–3.11's reconciliation and returning the correctly classified outcome — re-raise (never swallow) any other constraint's violation. Never expose the constraint name or raw SQL text to the caller.
- [ ] 3.14 `ALTER FUNCTION ... OWNER TO postgres`; `REVOKE ALL ... FROM PUBLIC`, `REVOKE ALL ... FROM anon`, `REVOKE ALL ... FROM authenticated`; `GRANT EXECUTE ... TO service_role` only. No internal `current_app_role()` check (design.md §4 — the Edge Function's own authorization, performed before this RPC is ever called, is the sole boundary; `auth.uid()` would resolve to `NULL` under the service-role connection this RPC is now reached through).
- [ ] 3.15 Add pgTAP tests: owner/`prosecdef`/`search_path` assertions; `PUBLIC`/`anon`/`authenticated` all lack `EXECUTE`, `service_role` has it; `created` on a clean first call; `already_completed` when the worker's raw email and the linked Auth user's email already both canonically equal the requested target at claim time, confirming the new row is inserted directly as `completed`; `resumed` when called again with an identical target while the first claim is still `active`, confirming exactly one row exists; `different_target_in_progress` when called with a different target while `active`, confirming the original row's target is unchanged and no second row was created; `manual_attention_blocking` when an existing row is `manual_attention_required`, regardless of the newly requested target, confirming no second row is created; `created` again for the same worker once its prior operation reached `completed`.
- [ ] 3.16 **(Finding: pre-claim rejection tests.)** `duplicate_worker_email` returned, with no operation row created at all, when another worker's `workers.email` already canonically matches the requested target — confirm via a row-count check before/after. `linked_auth_user_missing` returned, with no operation row created, when the resolved linked Auth user id does not exist in `auth.users` (construct this by deleting the `auth.users` row for a worker's linked profile inside the test transaction before calling the RPC).
- [ ] 3.17 **(Finding: ambiguity reconciliation tests, replacing the previously-incorrect assumption that ambiguity always inserts a new row.)** One existing `active` row whose stored `linked_auth_user_id` is made to disagree with a fresh resolution (e.g. by directly updating the row's `linked_auth_user_id` in the test setup) → confirm the *existing* row is transitioned to `manual_attention_required` in place, confirm no second row exists, confirm the returned `operation_id` equals the existing row's id; two separately-constructed blocking rows, one reachable only via `worker_id`, a different one reachable only via `linked_auth_user_id` for the same request → confirm `ambiguous_claim_state` is returned, confirm `operation_id` is `NULL`, confirm **neither** row was mutated; an existing `manual_attention_required` row found consistently by both searches → confirm `manual_attention_blocking`, confirm the row is untouched.
- [ ] 3.18 A direct test simulating the partial-index race (two near-simultaneous inserts) confirming the second caller receives a correctly classified `resumed`/`different_target_in_progress`/`manual_attention_blocking` outcome rather than an unhandled constraint-violation error; confirm an unrelated unique-violation (a deliberately-triggered one, unrelated to this table) is still re-raised, not swallowed.
