## Why

A worker's Supabase Auth login account is provisioned once, using whatever email was in `workers.email` at that time (`create-worker-account`, `link_worker_account`). If a worker's real-world email later changes (or was entered incorrectly), `workers.email` can be corrected through the ordinary worker-edit form, but the linked Auth account's login email is never updated — Supabase Auth and the `workers` table are two independent systems and nothing today keeps them in sync after initial provisioning. Today the only way to reach the same outcome is a manual, undocumented, Supabase-Studio-side Auth edit, with no synchronization back to `workers.email`.

## What Changes

- Add a new administrator-only per-worker action, **Actualizar correo de acceso**, for a worker who already has a linked self-service account. This action does exactly three things: update the existing linked Auth user's email, synchronize `workers.email`, and verify both converge on the requested address. **It never sends an access-link email.** Sending a fresh link remains the existing, separate "Reenviar enlace de acceso" action, which the administrator triggers explicitly afterward — an intentional two-step workflow (update, then resend), not one combined action.
- Add a new Edge Function (`update-worker-access-email`) that: authenticates and authorizes the caller before parsing any privileged input; creates or resumes a **durable database claim** for the correction using only the worker's id and the raw requested email (never a browser-supplied Auth user id, "current email," or profile id — every other value is resolved server-side); detects zero/one/multiple canonical Auth matches before mutating anything; updates the existing linked Auth user via `admin.updateUserById` (never creating or deleting an Auth user); synchronizes `workers.email` through a guarded RPC bound only to that claim; and returns a closed, delivery-free response describing whether the two systems now agree.
- Add a new internal table, `worker_access_email_corrections`, with exactly three states — `active`, `completed`, `manual_attention_required` — and partial unique indexes enforcing **at most one blocking (active or manual-attention) correction per worker and per linked Auth identity**. A `completed` operation never blocks a future correction; a `manual_attention_required` operation blocks indefinitely and has no in-app resolution path in this change (resolving it is a manual, technical task outside this feature's scope).
- Add a claim-resolution RPC, `claim_worker_access_email_correction(worker_id, requested_email)`, that accepts **only** these two trusted business inputs — it resolves the linked profile, the linked Auth user, and the canonical form of the requested email entirely server-side, rejects a duplicate worker email or a missing linked Auth user *before* creating any claim, and reconciles any existing blocking operation by inspecting it (never by blindly inserting a row that would violate the table's own uniqueness guarantees) — transitioning an inconsistent existing row in place when one is implicated, rather than ever attempting an impossible insert.
- Add a read-only, internal operation-context RPC, `get_worker_access_email_correction_context(operation_id)`, that the correction Edge Function calls immediately after claiming or resuming an operation, so every later step in that same request works from one single, operation-bound view of the worker/Auth identity rather than risking a second, independently-re-resolved (and potentially divergent) one.
- Add a worker-synchronization RPC, `sync_worker_email_after_access_correction(operation_id)`, bound **only** to the operation id — every other value (worker id, linked Auth id, requested email, and the immutable raw "expected worker email" concurrency guard) is loaded from the durable operation row itself, never re-supplied by the caller.
- Add two narrow, guarded transition RPCs (`mark_worker_access_email_correction_completed`/`..._manual_attention`), sharing one locking-and-linkage-revalidation protocol with the sync RPC, and restricting the manual-attention transition to a narrow, explicit set of reason codes — no code path permits a broad, direct mutation of the operations table.
- **All five of these internal RPCs, plus the existing canonical-email lookup RPC, are reachable only through the server's own service-role connection** — none is grantable to, or callable by, any authenticated browser session, regardless of role. The Edge Functions' own administrator authorization, performed once before any of these RPCs is called, is the sole authorization boundary; none of the RPCs performs its own redundant (and, under a service-role connection, meaningless) internal role check.
- Add a second, small, dedicated administrator-only Edge Function, `get-worker-access-email-context` (a browser-facing endpoint, distinct from the internal `get_worker_access_email_correction_context` RPC above, despite the similar name), solely to supply the confirmation dialog's masked (explicitly revealable) current-Auth-login-email display, with its own closed success/error response contract.
- Whenever an operation starts or resumes, freshly re-read the linked profile, current Auth email, and current `workers.email` before deciding anything; every uncertain external result triggers a fresh read before concluding anything is genuinely uncertain — `emailSynchronized` is derived only from what is actually observed.
- Define a closed, discriminated response contract — `status`, `reasonCode`, `retryable`, `emailSynchronized`, an optional display-only `message` — with **no delivery-related field of any kind**.
- Add a dedicated, accessible confirmation dialog (not a reuse of the existing shared `Modal.tsx`, which lacks dialog semantics, a focus trap, initial focus, focus restoration, and Escape handling) showing the worker's name, current `workers.email`, a masked (explicitly revealable) current Auth login email, the requested new email, and a warning that the login email will change. On success, it shows `"Correo de acceso actualizado. Envía ahora un nuevo enlace de acceso al trabajador."` with a clear (non-automatic) path to the existing resend action.
- Add pgTAP coverage for the new table/RPCs (owner, grants, `search_path`, claim create/resume/conflict/manual-attention-blocking behavior, partial-unique-index race reconciliation, all closed result states) and Deno unit/contract tests for both new Edge Functions (every reconciliation branch, authorization-before-parsing, transport uncertainty with fresh-read reconciliation, response-minimization for the context endpoint, and explicit assertions that no delivery mechanism is ever invoked).
- Manual local verification covering actual local runtime Auth behavior; hosted/production behavior remains a separate, human-owned verification task — no remote Supabase changes are performed by this change.

**BREAKING**: None. This is a new, additive action; no existing route, table name, RPC signature, or Edge Function contract changes. The existing `resend-worker-access-link` implementation is not modified by this change.

## Capabilities

### New Capabilities
- `worker-access-email-correction`: administrator-only correction of a worker's linked Supabase Auth login email after initial account provisioning, keeping `workers.email` and the Auth account synchronized, without ever creating a second Auth account, deleting the existing one, or sending an access-link email as a side effect of the correction itself.

### Modified Capabilities
(none — `worker-password-recovery` and the existing worker/profile linking and resend-access behavior are unchanged; this change adds a new, separate action rather than altering any existing requirement)

## Impact

- **New migrations**: `worker_access_email_corrections` (new internal table, two partial unique indexes covering `active` **and** `manual_attention_required`, check constraints on `requested_canonical_email` and `last_reason_code`); a server-only canonical-Auth-email-match lookup RPC; `claim_worker_access_email_correction` (trusted-input-only, with pre-claim duplicate/missing-Auth rejection and non-destructive ambiguity reconciliation); `get_worker_access_email_correction_context` (read-only, internal); `sync_worker_email_after_access_correction` (operation-id-only); two narrow guarded transition RPCs sharing one locking/linkage-revalidation protocol. All six RPCs: `SECURITY DEFINER`, explicit owner, fixed empty `search_path`, a single fixed two-key advisory-lock order reused identically wherever locking is needed, and — for the five newly added in this change — `EXECUTE` granted to `service_role` only, revoked from `PUBLIC`/`anon`/`authenticated`.
- **New Edge Functions**: `supabase/functions/update-worker-access-email/` and `supabase/functions/get-worker-access-email-context/` (both registered in `supabase/config.toml` with `verify_jwt = true`). Neither uses an anon-only client or calls any email-sending API — `update-worker-access-email` has no reason to hold one.
- **New frontend files**: a service function in `src/services/apiProfiles.ts` for each endpoint, a mutation hook, a query hook for the masked current-Auth-email context, and a new dedicated dialog component (not `Modal.tsx`) with its own focus/ARIA/Escape/pending-guard handling; a new per-worker action added to `WorkerRow.tsx`'s existing `Menus` list, gated on `isAdmin && isLinked`.
- **Query invalidation**: `["workers"]`, `["profile"]`, `["linked-worker-accounts"]`, plus a new query for the masked current linked Auth email — invalidated on every outcome that could have changed state, never on one rejected before any side effect.
- **No changes** to `workers`/`profiles` schema, no uniqueness constraint on `workers.email`, no changes to `create-worker-account`, `link_worker_account`, or `src/ui/Modal.tsx`. **No changes to `resend-worker-access-link`** unless a minimal compatibility adjustment proves strictly necessary, in which case it is documented and justified explicitly, not made silently.
- **Tests**: new pgTAP file(s) covering claim create/resume/different-target-conflict/manual-attention-blocking/ambiguous-state coverage, partial-unique-index race reconciliation, and both transition RPCs; new Deno test file(s) for both Edge Functions, including explicit assertions that no delivery mechanism of any kind is ever invoked.
- **Pre-merge reconciliation task**: if the still-unmerged `harden-and-bulk-provision-worker-accounts` branch merges before this change, its own canonical-Auth-email-lookup migration must be reconciled with this change's (reuse rather than duplicate) before either merges.
- **No remote/hosted Supabase operations** are performed by this change at any point; hosted behavior remains a separate, human-owned verification step.
