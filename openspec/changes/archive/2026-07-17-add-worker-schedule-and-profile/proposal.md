## Why

The worker self-service portal (`WorkerAppLayout`) currently offers only `Mis documentos`, with no navigation at all — a teacher who wants to see their own weekly schedule or confirm their own basic record on file (name, contact info, role) has no way to do so and must ask an administrator. The underlying data already exists (`schedule_assignments`, `schedule_teachers`, `workers`) and the admin side already has a full scheduling module, but nothing exposes a *read-only, ownership-scoped* view of it to the worker who owns it — and the two schedule tables currently have no row-level ownership enforcement at all (`USING (true)` for any authenticated user), so a migration is required before any worker-facing schedule feature can be built safely.

## What Changes

- Add `/my-schedule` ("Mi horario"): a read-only schedule view combining the worker's own `schedule_assignments` (class assignments) and `schedule_teachers` (other scheduled activities) for a selected semester, normalized into one explicit `WorkerScheduleEntry[]` contract where **every authorized row is always represented — none are ever discarded for having a malformed or missing weekday/time**. Consumed by two presentations of the same data — a weekly timetable/grid on desktop, visually aligned with the administrator schedule's institutional tokens, and a day-grouped agenda on mobile, switching at the existing `900px` breakpoint — plus a shared, clearly labeled "Horario no especificado" section on both, showing the actual entries (never just a count) that can't be placed in the normal grid/agenda because their time or weekday is missing, malformed, or (desktop only) not aligned to a configured canonical block. Defaults to the chronologically latest *valid* semester (a new deterministic comparator, not `semesters[0]` and not the existing `findLatestSemester`'s skip-malformed behavior, which isn't sufficient for a selector that must also handle an all-malformed semester list deterministically), with a selector to inspect historical semesters.
- Add `/my-profile` ("Mi información"): a read-only view of an explicit, safe subset of the authenticated worker's own `workers` row (name, email, phone, worker type, status, specialty, function performed, profile picture) — never the full row, never RFC/address/observations/timestamps/raw IDs. Documented honestly as request/UI minimization, not a database-level column confidentiality guarantee (see Impact).
- Add worker self-service navigation to `WorkerAppLayout` (currently has none): `Mis documentos`, `Mi horario`, `Mi información`, as `NavLink`s with `aria-current="page"` on the active route, positioned as its own row beneath the shared header so it never conflicts with the header's account popover/theme toggle.
- Extend `src/ui/routeContext.ts`'s pattern list with `/my-schedule` → "Mi horario" and `/my-profile` → "Mi información", preserving most-specific-first matching and the existing `ENUB` fallback.
- Add a single `WorkerRouteGate` (mirroring the admin `RoleGate.tsx`) wrapping the entire worker route branch above `WorkerAppLayout` — not a per-page component any page imports or adopts. All three worker self-service pages (`MyDocuments.tsx`, `MySchedule.tsx`, `MyProfile.tsx`) render beneath it in the route tree, so `WorkerAppLayout`, `Header`, `WorkerNav`, and every page's content only ever mount after the gate has resolved; none of the three pages contains its own authorization logic. This removes `MyDocuments.tsx`'s prior inline gating (staff/admin redirect, invalid-link redirect) rather than duplicating it a third time or refactoring the page to hold its own copy of it. `/pending-access` — the gate's own denial target — is a sibling route tree, outside the worker layout, under its own `PendingAccessLayout`.
- **Migration required, and not merely additive**: replace the current fully-open `schedule_assignments`/`schedule_teachers` read policy with the staff/admin-unrestricted + worker-own-row (`worker_id = current_worker_id()`) pattern already established for `workers` and `worker_documents` — deployed as one transaction with explicit precondition checks (verifying the exact policy being removed, and detecting any other permissive policy that could preserve global access under a different name) and postcondition checks that verify each new policy individually by name on every dimension (`cmd`, `permissive`, exact `roles`, and a normalized predicate captured empirically from Postgres itself, not hand-authored), plus no anonymous access and an exact final policy catalog — failing closed on any mismatch, inside the migration transaction itself, not only in a test suite run afterward.
- **Write-policy behavior is proven unchanged, not merely assumed**: since the migration touches `SELECT` only, structural and behavioral tests confirm `INSERT`/`UPDATE`/`DELETE` on both tables are byte-for-byte and behaviorally identical after the migration — six behavioral write cases in total (three commands × two tables) — without endorsing those write policies as secure (see Fixed Follow-ups #3).
- **New, narrow, worker-specific schedule queries** — the schedule page does **not** reuse the existing admin `getScheduleAssignments`/`getScheduleTeachers`, since `getScheduleTeachers()` embeds the full `workers(*)` row on every request; two new functions with an explicit column projection (no worker embed at all, no RFC/address/observations/timestamps) are added instead.
- No RLS change needed for `workers`: "Workers can read own worker row" already exists and is correct; the profile page reads it via a new, explicitly-column-limited service function (same pattern as the existing `getWorkerIdentityById`), not the admin `getWorkerById()`'s `select("*")` — documented as limiting what this application requests, not as a database-enforced column boundary, since `workers` RLS authorizes the row, not individual columns.
- **BREAKING**: none. No existing route, component prop, or query shape changes for any current caller; the admin `getScheduleAssignments`/`getScheduleTeachers` are entirely unchanged and untouched by this proposal.

## Capabilities

### New Capabilities
- `worker-self-service-navigation`: the `Mis documentos` / `Mi horario` / `Mi información` navigation added to the worker portal layout, and the corresponding route-context labels.
- `worker-schedule-viewing`: the read-only "Mi horario" page — semester selection/ordering, the worker-specific query layer, the normalized schedule contract, the desktop grid and mobile agenda presentations, and all loading/empty/error states.
- `worker-profile-viewing`: the read-only "Mi información" page — the explicit field allow-list, exact status/placeholder mappings, its presentation, and all loading/empty/error states.
- `worker-owned-data-access`: the authorization/RLS backbone shared by both pages — the database-level, verified-deployment ownership guarantee that a worker can only ever read data belonging to the worker linked to their own authenticated profile, independent of any client-supplied id.

### Modified Capabilities
_None — `schedule-semester-scoped-queries`, `semester-generation`, `teacher-schedule-canonical-blocks`, and `scholar-schedule-canonical-blocks` are relied upon as-is; none of their requirements change. This change only adds new, additional, worker-specific consumers alongside the unmodified admin query layer, and tightens row-level access beneath both._

## Impact

- **Code**: new `src/pages/MySchedule.tsx`, `src/pages/MyProfile.tsx`, new `src/ui/WorkerRouteGate.tsx` (a branch-level route gate wrapping the entire worker route group above `WorkerAppLayout` in `src/App.tsx`, with `MyDocuments.tsx`'s prior inline gating removed rather than duplicated), new normalized-schedule/comparator/formatting pure modules under `src/features/schedules/`, new desktop-grid and mobile-agenda presentational components (new, not shared code with admin CRUD components — see design.md §8), a new `WorkerNav.tsx` added to `src/ui/WorkerAppLayout.tsx`, a new `apiWorkerSchedule.ts` service (worker-specific, narrow-projection queries), a new `getMyWorkerProfile` in `src/services/apiWorkers.ts`, `src/ui/routeContext.ts` (two new pattern entries), `src/App.tsx` (two new routes plus the branch-level gate wiring).
- **Data**: two new narrow-projection schedule queries (no `workers` embed) and one new narrow-projection profile query. The existing admin `getScheduleAssignments`/`getScheduleTeachers`/`getWorkerById` are entirely unchanged. No new tables, no new columns.
- **Database**: one migration replacing `schedule_assignments`/`schedule_teachers` SELECT RLS, deployed with explicit pre/postcondition verification (design.md §7) — not a plain additive change, since it removes a currently-effective open-read grant. No change to `workers`, `profiles`, `worker_documents`, `semesters`, `subjects`, `groups`, or `degrees` RLS in this change (see Fixed Follow-ups below for the broader catalog-table gap).
- **Dependencies**: none added.
- **Routes**: two new routes (`my-schedule`, `my-profile`) added under the existing worker `<Route>` group in `App.tsx`. No existing route changes.

## Fixed Follow-ups (definite, separate changes)

Identified during this change's investigation, deliberately out of this change's scope, and recorded here as follow-ups rather than left as open questions:

1. Harden the remaining anonymously-readable catalog tables (`semesters`, `subjects`, `groups`, `degrees`, and any other table still on the base-schema `USING (true)` policy with no `TO authenticated` clause).
2. Fix `WorkerDocumentsView.tsx`'s unordered `semesters[0]` default-semester selection — the exact anti-pattern this proposal's own semester-selection design avoids. **Not modified in this change.**
3. Audit and harden the broad, unrestricted INSERT/UPDATE/DELETE policies on `schedule_assignments`/`schedule_teachers` — this change's migration tightens `SELECT` only.
4. Design true column-level worker-profile confidentiality (a restricted view or `SECURITY INVOKER` function with a database-enforced column allow-list) — this change's `getMyWorkerProfile` projection is application-level minimization only, not a database-enforced guarantee, since current `workers` RLS authorizes a worker's own row in full.
5. Extract a single shared recess-block constant used by both the worker and admin schedule presentations (design.md §18) — the admin grid and PDF exports each independently hard-code the same two recess times/label today, with no shared constant even among themselves; this change's worker-schedule-scoped `schoolDayBlocks.ts` deliberately does not retrofit them, to avoid touching shipped, tested admin schedule/PDF code.
